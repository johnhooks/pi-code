# Code Workflow

Use this workflow when the task is accepted and ready to change code.

Implementation should start with a plan and stop for approval before editing.
The plan is not sacred. If the code disproves an assumption, adjust the approach
and make the deviation visible.

## Expectations

Start from a task, ticket, issue, brief, or developer instruction. Inspect
enough code to understand the seam before editing. Present a concise plan and
wait for approval.

Keep changes inside the approved scope. Make the smallest coherent set of
changes that satisfies the task. When implementation edits are complete,
follow-up review, cleanup, and verification responsibilities run before final
reporting.

Prefer prose over long checklists when reporting approach and results. Use lists
only when they make changed items, verification, or risks easier to scan.

## Done Means

The approved scope is implemented. The diff has been reviewed against the task
and active conventions, one cleanup pass is complete, and relevant checks have
run or the reason they were not run is clear. Any deviations from the approved
plan are summarized.
