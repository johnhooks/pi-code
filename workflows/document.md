# Document Workflow

Use this workflow when writing or updating project documentation.

Documentation should reflect the current implementation unless it is explicitly
describing planned work. It should help the reader accomplish something or
understand a decision without requiring them to reverse-engineer the code.

## Expectations

Identify the audience and purpose before writing. Read the relevant source,
configuration, scripts, or existing docs before making claims. Keep instructions
direct and current.

Prefer concise prose and examples over exhaustive background. Use Mermaid
diagrams for flows, sequences, state transitions, or architecture relationships
when a visual structure helps. Mark planned behavior as planned, not current.

## Done Means

The documentation matches the current implementation or clearly labels planned
behavior. Commands, paths, and links have been checked when practical. Diagrams
are simple enough to maintain. The changed docs are scoped to the requested
topic.
